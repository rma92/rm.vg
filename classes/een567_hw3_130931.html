  1. (15 points) [Exercise 3.2] Compare and contrast the two-tier client-server architecture for traditional DBMSs with the three-tier client-server architecture. Why is the latter architecture more appropriate for the Web?

    A determination between layers, and tiers should be mentioned beforehand. Layers are a way to logically break code into components. Tiers are physical nodes on which to place the aforementioned components.

    A two-layer architecture contains two layers, the Data layer, and the layer that handles both Logic and Presentation. These layers can be distributed onto multiple machines to make a two-tier system.

    A three-layer architecture contains three layers, the Presentation layer, the Logic layer, and the Data layer. These layers can be distributed onto multiple machines to make a two-tier, or three-tier system. This system separates the logic from the storage and presentation layers, allowing it to run on an independent machine, if desired.

    In the case of the web, a three-tier, three-layer system is extremely desirable. In a web application, the presentation layer will likely be considered the web browser of the user, while the web server running some application (written in PHP, ASP.NET, Java Server Pages, etc.) which accesses the database be the logic layer. The Data layer/tier would be the actual database. In a web application, this more distributed model is preferable, as it allows for better security (reducing the chances of Denial of Service attacks), and better load balancing.

  2. (15 points) [Exercise 3.11] Describe the internal architecture of Oracle.

    An Oracle database instance is contained primarily within a single process acting as a system service. The process contains the memory structures and threads that are needed to access the database.

    When a client connects to the instance, it connects via a shadow thread. The shadow thread then connects directly with the process monitor. The memory structures (within the process) include a shared pool of library cache, and a data dictionary cache, a database buffer cache, and a redo log buffer.

    Other threads running within the process include the required database writer, log writer, process monitor, and system monitor. Additionally, the optional features checkpoint process, archive process, and distributed recovery process run as threads within the process.

    There is also an internal Oracle Database containing data files, control files, and redo log files. Additional data associated with this database is the parameter file, password file, and archived log files.

  3. (15 points) [Exercise 4.5] Discuss the differences between the candidate keys and the primary key of a relation. Explain what is meant by a foreign key. How do foreign keys of relations relate to candidate keys? Give examples to illustrate your answer.

    A superkey is the set of attributes of a relation variable for which it holds that in all relations assigned to that variable, there are no distinct tuples that have the same values for these attributes. That is, the superkey can be used to select exactly one row in the database in a reliable and consistent manner.

    A candidate key is a minimal superkey for a particular relation; a set of attributes for which no two tuples contain the same values for these attributes, and there is no proper subset of these attributes that will adequately allow for uniqueness (i.e. the set is minimal). There may be multiple candidate keys for a particular relation.

    A primary key is a candidate key that is selected as the main index into the relation. There may only be one primary key in a particular relation. The remaining candidate keys are called alternate keys. It is preferred to select the primary key that is most efficient to compare when performing an action against the database (e.g. integers are preferred over strings or blobs, as the former can be compared in significantly less time than the latter).

    A foreign key is a field in a relation that refers to records in another relation. For example, we can have a table of high schools, which contains the address and city of a particular school, as well as a state-assigned ID, which will be used as the primary key. Another table contains a list of all students in the state. This table contains the student's first and last name, student ID (primary key), and the high-school ID (a foreign key). In this case, if one wishes to look up the city in which a student goes to school, he can perform a join between the two tables, and get this information by selecting data from the table of high schools as well as the table of students.
